# 小项目：DeskFlow — 工位预订 API

**主题：** 帮助芬兰办公室管理共享工位。员工需要一种简单方式来查看哪些工位空闲，并为某一天预订工位到办公室办公。

**目标：** 小组协作构建一个由数据库支持的小型 Java REST API，并在 **8~10 分钟** 内完成展示。

**时间盒：** 约 5 小时开发 + 展示  
**团队规模：** 4–5 人  
**AI 工具：** 允许且建议使用，用来提高效率；但你们必须真正理解并能够解释自己实现的每个接口，并准备回答最后的提问。

今天结束前，你们团队应具备：

## 领域模型（规定项）

至少使用以下概念。列名可以不同，但语义必须一致。

### `desk`

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `code` | 唯一短编码，例如 `KRK-3F-12` |
| `floor` | 楼层号，整数 |
| `has_monitor` | 布尔值 |
| `is_active` | 布尔值，停用工位不能被预订 |

### `booking`

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `desk_id` | 外键，关联到 `desk` |
| `employee_name` | 字符串 |
| `date` | 预订日期，一条记录对应一天 |
| `created_at` | 该预订创建时的时间戳 |

**核心规则：** 同一个工位在同一天 **最多只能有一条预订记录**。

至少准备 **8 个工位**，分布在 **2 个以上楼层**，并准备几条示例预订数据。

## 必需 API（请按这些契约实现）

实现以下接口，确保演示时行为清晰且一致。

### 1. 健康检查

```http
GET /api/health
```

**200** 响应示例：

```json
{ "status": "ok" }
```

### 2. 查询工位列表

```http
GET /api/desks
```

可选查询参数（至少实现一个）：

- `floor` — 按楼层过滤
- `hasMonitor` — `true` / `false`

**200** — 返回工位数组（包含 `id`、`code`、`floor`、`hasMonitor`、`active`）。

### 3. 创建预订

```http
POST /api/bookings
Content-Type: application/json
```

请求体：

```json
{
  "deskId": 1,
  "employeeName": "Anna Kowalska",
  "date": "2026-07-24"
}
```

行为要求：

- 成功时返回 **201** + 新建预订的 JSON
- 工位不存在时返回 **404**
- 工位未启用、日期缺失/非法、或 `employeeName` 为空白时返回 **400**
- 同一工位在同一天已被预订时返回 **409**

### 4. 查询某天的预订

```http
GET /api/bookings?date=2026-07-24
```

**200** — 返回该日期的预订数组（包含 desk `code` 或 `deskId`，便于演示阅读）。

### 5. 取消预订

```http
DELETE /api/bookings/{id}
```

- 删除成功时返回 **204**（或 **200** 并附带简短消息）
- 预订 id 不存在时返回 **404**

## 创意部分（必做，可加分，开放设计）

实现 **一个或多个** 符合 DeskFlow 主题的额外 API 功能。由你们团队自行设计接口、请求/响应结构以及业务规则。

可以选择一些真正有用的方向，例如：

- 可用性查询：某一天哪些工位空闲
- 员工视角：查询“我”的所有预订
- 工位搜索：按编码前缀或关键字搜索
- 软规则：限制最多提前 N 天预订
- 统计：某一天每层楼的预订数量
- 候补 / “有人取消时通知我”（可简化成内存实现）

**额外加分：** 做一个小型 **前端**（哪怕只有一个 HTML 页面）调用你们的 API，例如按日期列出空闲工位并创建预订。可选，仅在必需 API 和创意接口完成后再做。
